# 架构说明

## 1. 整体结构

本工程使用 LangGraph 管理研发流程，每个阶段是一个节点：

```text
init
  -> retrieval
  -> requirement_review
  -> requirement_gate
  -> architecture_review
  -> architecture_gate
  -> detailed_design
  -> coding
  -> test_design
  -> test_execution
  -> closed
```

其中 `requirement_gate` 和 `architecture_gate` 是 Human in the loop 节点。

## 2. Agent 协同

| Agent 节点 | 职责 |
| --- | --- |
| retrieval_agent | 调用 TaskPath、文档知识图谱、CodeGraph，构建上下文包 |
| requirement_agent | 将用户输入拆成可确认的需求项 |
| architecture_agent | 将需求映射到模块、接口、依赖和架构决策 |
| detailed_design_agent | 具体到文件、函数、宏、变量、结构体和操作步骤 |
| coding_agent | 执行或计划代码修改 |
| test_design_agent | 使用测试用例 Skill 思路生成三层测试用例 |
| test_execution_agent | 执行 dry-run/static 检查并生成三层验证报告 |
| close_agent | 生成最终关键成功路径并写入记忆库 |

## 3. 记忆管理

SQLite 数据库位于：

```text
memory/cswrite_agent.sqlite3
```

包含三张表：

| 表 | 用途 |
| --- | --- |
| runs | 保存每次运行最新状态，可用于人工确认后恢复 |
| events | 保存每个阶段事件，便于调试 |
| success_paths | 只保存最终关键成功路径，避免半成品污染长期记忆 |

## 4. MCP 复用方式

`mcp_adapter.py` 通过 stdio 启动现有 MCP server，不依赖 OpenCode：

| MCP | 用途 |
| --- | --- |
| taskpath_memory | 检索历史成功路径，最终保存关键路径 |
| kg_rag_neo4j | 检索 Neo4j 文档知识图谱 |
| codegraph | 检索代码结构、符号和参考实现 |

## 5. 三层验证

测试执行不能只验证编码。必须按顺序输出：

1. `detail_design_verification`：验证代码修改和详细设计项。
2. `architecture_verification`：验证人工确认的架构决策是否落地。
3. `requirement_acceptance`：验证用户确认的需求和验收标准是否满足。

缺少任何一层时，只能算部分验证，不能保存最终成功路径。
