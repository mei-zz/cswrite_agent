# CSWrite LangGraph Agent 面试演示指南

这份指南用于现场讲解和真实跑通 demo。建议按“状态机、工具、记忆、人工确认、测试闭环”五条主线展开。

## 1. 环境检查

```powershell
$env:PYTHONPATH="src"
D:/anaconda3/python.exe -m cswrite_langgraph_agent.cli doctor
```

重点说明：

- `llm_api_key_present=True` 表示真实 API key 已被配置层读取，但不会打印密钥内容。
- `code_root/kg_script/codegraph_script/taskpath_script` 用来证明 real 模式可接入真实工程和 MCP。

## 2. 导出 LangGraph 状态图

```powershell
$env:PYTHONPATH="src"
D:/anaconda3/python.exe -m cswrite_langgraph_agent.cli graph --output docs/langgraph_flow.mmd
```

讲解顺序：

- `task_analysis`：自然语言任务结构化，生成 `chip_spec`。
- `planner`：根据任务复杂度制定执行计划、自修复预算和条件节点策略。
- `retrieval`：demo 读本地文档和示例代码，real 模式走 MCP，并通过 `context_manager` 生成受控上下文。
- `requirement_gate` / `architecture_gate`：人工确认点。
- `coding`：按详细设计修改示例仓库或真实仓库。
- `test_design` / `test_execution`：三层验证。
- `critic`：检查需求、设计、代码和测试之间是否存在矛盾。
- `reflection`：当 Critic 判定可修复时，总结失败原因并决定回退到哪个节点。
- `traceability_matrix`：生成需求、架构、详细设计、代码、测试结果的追溯矩阵。
- `observability`：生成节点耗时、LLM 调用、上下文覆盖、记忆计数报告。
- `closed`：只有成功闭环才写入 SQLite 成功路径记忆。

## 4. 跑真实百炼模型闭环

```powershell
$env:PYTHONPATH="src"
D:/anaconda3/python.exe -m cswrite_langgraph_agent.cli run `
  --task "新增一个测试用 8 位旧 ICP 芯片 VMODEL_BAILIAN_8BIT，验证完整研发流程" `
  --mode demo `
  --llm-mode bailian `
  --auto-approve `
  --execute-code
```

讲解重点：

- 结构化字段由代码保证稳定，模型负责阶段判断、风险提示和总结。
- `state_latest.json` 中的 `llm_calls` 可以证明每个核心阶段都真实调用模型。
- 失败的模型调用会进入 `errors`，不会中断可重复的工程流程。

## 5. 演示自修复

```powershell
$env:PYTHONPATH="src"
D:/anaconda3/python.exe -m cswrite_langgraph_agent.cli run `
  --task "新增一个测试用 8 位旧 ICP 芯片 VMODEL_REPAIR_8BIT，验证自修复流程" `
  --mode demo `
  --llm-mode bailian `
  --auto-approve `
  --execute-code `
  --inject-coding-defect `
  --max-repair-attempts 1
```

这条命令会故意让首轮编码漏掉注册表写入。讲解重点：

- `06_test_execution.md` 首轮发现注册表缺失。
- `10_critic_report.md` 判断问题属于代码/测试一致性缺口，建议回退 `coding`。
- `11_reflection_attempt_1.md` 总结失败原因并消耗一次修复预算。
- 第二轮 `coding -> test_design -> test_execution -> critic` 后通过，最终 `closed/success`。

## 7. 面试讲法

- 不是让 LLM 直接改代码，而是“结构化状态 + 可审计产物 + 确定性执行”。
- 每个节点结束都会持久化 SQLite 和 `state_latest.json`，所以流程可恢复、可追踪。
- MCP 被封装在 `Retriever` 里，demo/real 只切换检索后端，不改变 V 模型主流程。
- 上下文管理不是简单拼接文本，而是做预算、评分、入选证据、覆盖度和冲突提示。
- 记忆分层为 `episodic/semantic/procedural/negative`，成功路径和失败路径分开沉淀。
- Planner/Critic/Reflection 让流程具备“计划、审查、反思、修复”的 agent 控制闭环。
- 只有最终三层验证和追溯矩阵都通过，才把关键成功路径写入 `success_paths`，避免把失败经验当成功经验复用。
